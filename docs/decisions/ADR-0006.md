# ADR-0006 — Périmètre fonctionnel V1 et roadmap V2+

- **Date** : 2026-05-03 (esquisse) / TBD (finalisation J0)
- **Statut** : draft
- **Source brief** : D5.1, D5.2, D8.1, D8.2, §8 (Périmètre V1)

## Contexte

Holmes MCP V1 expose une box Jeedom à des LLMs. La question centrale : faut-il inclure des capacités d'écriture (lancer un scénario, activer/désactiver un équipement, écrire dans le dataStore) dès V1 ? Le projet jeedom-audit (ADR-0006 sources) maintient la lecture seule perpétuelle. Holmes MCP, lui, a vocation à évoluer vers l'écriture — mais pas en V1.

## Décision pressentie

**V1 : lecture seule + assistance à la conception.**

Aucune action modifiante en V1. Les capacités d'écriture (lancer scénario, activer/désactiver, écrire variable dataStore, exécuter commande action) sont **candidates roadmap V2+** (via API JSON-RPC uniquement, jamais via SQL — ADR-0005).

**Assistance à la conception V1** : émergente côté LLM, pas de tool serveur dédié. Le LLM lit l'installation via les tools de lecture, comprend le contexte, et rédige lui-même les propositions de scénarios ou de configuration en langage naturel. Les tools de lecture doivent retourner des structures riches et lisibles (conventions Jeedom représentées : Types Génériques, `#[O][E][C]#`, `logicalId`).

**Discipline anti-drift** (D8.2) : aucune feature non tranchée dans le brief ne rentre en V1. Tout nouveau besoin → ROADMAP.md. Si bloquant pour V1 → ADR + validation PO explicite.

## Conséquences pressenties

- Confiance utilisateur maximale (la V1 ne peut pas modifier l'installation)
- Asymétrie de risque favorable : V2 ajoute l'écriture sans rien casser ; une V1 avec écriture qui rate est difficile à reconstruire en termes de confiance
- Les descriptions des tools rappellent les conventions Jeedom pour guider le LLM dans ses propositions
- La bascule V1 → V2+ nécessitera des garde-fous explicites (scopes, confirmations, whitelist API, journal des actions)
