# ADR-0007 — Liste des 25 tools V1

- **Date** : 2026-05-03 (esquisse) / 2026-05-04 (proposed J3-3)
- **Statut** : proposed
- **Source brief** : D5.3, D5.4, D5.5, D5.6, D5.6.bis, D5.7

## Contexte

Les tools MCP sont les verbes appelés par le LLM pour interroger la box. La liste V1 doit couvrir l'ensemble des besoins d'audit et de compréhension d'une installation Jeedom, tout en étant suffisamment granulaire pour que le LLM choisisse le bon tool. La stabilité des schémas in/out est critique (D5.5) pour la compatibilité avec jeedom-audit futur consommateur MCP.

## Décision pressentie

**25 tools en 6 familles** (liste figée en V1, sauf raffinements internes validés PO) :

**Famille 1 — Découverte d'install** : `get_install_overview`, `list_objects`, `list_plugins`, `get_config`

**Famille 2 — Équipements et commandes** : `list_equipments`, `find_equipments_advanced`, `get_equipment`, `find_equipment_by_name`, `list_commands`, `find_commands_advanced`, `get_command_history`

**Famille 3 — Scénarios** : `list_scenarios`, `find_scenarios_advanced`, `get_scenario`, `get_scenario_structure`, `describe_scenario`, `find_scenario_dependencies`, `get_scenario_log`

**Famille 4 — Variables / dataStore** : `list_datastore_variables`, `get_datastore_variable`

**Famille 5 — Logs et diagnostic** : `tail_log`, `list_log_files`, `get_health_summary`

**Famille 6 — Recherche transverse** : `search_text`

**Power-user / Audit** : `query_sql` (SELECT-only, blacklist tables sensibles, LIMIT obligatoire, sanitisation runtime, mini SQL cookbook dans la description)

**Politique semver des tools** (D5.5) :
- `1.0.x` : schémas inchangés
- `1.x.0` : nouveaux tools ou paramètres optionnels rétro-compatibles
- `2.0.0` : breaking changes possibles + migration guide

La liste a été raffinée en J1-2 via la matrice de couverture D5.8 (ADR-0019) : **0 ajout requis**. Les 25 tools couvrent intégralement les 13 WF jeedom-audit — liste figée pour V1.

**État d'implémentation au 2026-05-04 (J5-1)** : 24/25 tools livrés (F1+F2+F3+F4+F5+F6). Reste : `query_sql` (J5-2). La liste est confirmée stable.

## Divergences et déférés documentés (audit J3-5)

### Divergences mineures acceptées

**`get_config` — paramètre `plugin` vs `namespace` (brief D5.6.bis)**
Le brief nomme le paramètre `namespace`. L'implémentation utilise `plugin` (nom de colonne MySQL, plus clair pour le LLM). Le paramètre est optionnel (None ou `*` = tous les namespaces) conformément au brief. Schéma MCP exposé : `plugin`. Stable V1.x.

**`get_scenario` — log séparé en `get_scenario_log`**
Le brief D5.3 inclut "log dernier run" dans la description de `get_scenario`. L'implémentation sépare volontairement les deux : `get_scenario` retourne les métadonnées + état runtime, `get_scenario_log` retourne le contenu du log (jusqu'à 500 lignes). Séparation de concerns — le log peut être volumineux et n'est pas toujours utile. La description de `get_scenario` cross-référence `get_scenario_log`.

**`find_scenario_dependencies` — callers uniquement (vs "qui est appelé par lui" dans le brief)**
Le brief D5.3 décrit ce tool comme bidirectionnel. L'implémentation via `usage_graph.resolve('scenario', ...)` retourne uniquement les callers (qui appelle ce scénario). Les callees (scénarios appelés par ce scénario) sont accessibles via `get_scenario_structure(follow_scenario_calls=1+)`. Les WF jeedom-audit (WF12, WF13) combinent les deux tools — couverture intacte. Documentation corrigée.

**`get_command_history` — history récente via MySQL (D4bis.6 prévoyait API JSON-RPC)**
D4bis.6 route l'historique récent vers l'API JSON-RPC (`cmd::getHistory`). L'implémentation utilise MySQL (table `history`) pour les deux niveaux (récent + archivé). Justification : la table `history` est accessible en lecture seule, plus fiable (pas de dépendance apikey), validée sur box réelle (68 tests intégration). Évolution candidate V1.x : enrichissement optionnel via API si MySQL manque des aggrégations PHP.

### Filtres de `find_commands_advanced` (D5.3 : "valeur courante")

Le brief D5.3 mentionne "valeur courante" comme champ de `find_commands_advanced`. Non implémenté (J3-4bis) — l'enrichissement API (N×`eqLogic::fullById`) est prohibitif sur une liste transverse de commandes. Pour `currentValue` : résoudre l'`equipment_id` via `find_commands_advanced`, puis appeler `list_commands(equipment_id)` ou `get_equipment(equipment_id)`. Skill-coverage-matrix WF8 mis à jour (J3-5).

### Filtres manquants dans `find_equipments_advanced` (D5.3)

- **"dernier seen" (`lastCommunication`)** : champ PHP runtime stocké dans le blob `status` de `eqLogic`, non filtrable directement en SQL. Candidat V1.x : filtrage post-enrichissement API.
- **"historisation"** : `isHistorized` est un attribut de `cmd`, pas de `eqLogic`. Le brief est ambigu — le filtre est correctement implémenté dans `find_commands_advanced(is_historized=True)`. Pour les eqLogics avec au moins une commande historisée : utiliser `query_sql` (JOIN eqLogic + cmd).

### Filtres manquants dans `find_scenarios_advanced` (D5.3)

- **"dernière exécution"** : `lastLaunch` est runtime-only (absent de MySQL). Filtrage possible post-enrichissement API — candidat V1.x.
- **"contient telle commande"** : nécessite un JOIN sur `scenarioExpression WHERE expression LIKE '%#cmdId#%'`. Faisable mais complexe — candidat V1.x.
- **"mots-clés"** : partiellement couvert par `name_contains`. Recherche cross-fields (name + description + trigger) — candidat V1.x. Pour recherche transverse globale, utiliser `search_text` (F6, J5).

## Conséquences pressenties

- Chaque tool a un module Python dédié dans `resources/holmesMcpd/tools/`
- Les descriptions des tools rappellent les conventions Jeedom pour guider le LLM
- `query_sql` inclus dès V1 préserve la flexibilité pour les workflows d'audit avancé et la future consommation par jeedom-audit
- La stabilité des schémas V1.x protège jeedom-audit qui pourra exiger `holmesMcp >= 1.0, < 2.0`
