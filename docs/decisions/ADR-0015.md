# ADR-0015 — Modèle opérationnel PO / Claude Code

- **Date** : 2026-05-03 (esquisse) / TBD (finalisation J0)
- **Statut** : draft
- **Source brief** : §0 (Modèle opérationnel), §13 (Modèle opérationnel formalisé), D13.1-D13.10

## Contexte

Holmes MCP est développé par un binôme PO non-développeur / Claude Code. Sans contrat opérationnel explicite, le risque est une dérive progressive : questions ouvertes au lieu d'arbitrages structurés, sessions marathons, perte de continuité entre sessions espacées, gaspillage de cycles sur des détails triviaux. Le contrat opérationnel de jeedom-skills est adopté comme base, avec des ajustements pour l'accès SSH et l'automatisation accrue.

## Décision pressentie

Adoption du **contrat opérationnel jeedom-skills** (D13.1) avec les ajustements Holmes MCP :
- Claude Code dispose d'un accès SSH à la box (alias `Jeedom`, user `jeedom` — blacklist commandes D11.7)
- Plus de tâches techniques automatisées par Claude Code : setup initial plugin, tests live, fixtures réelles, logs daemon
- Goulets PO listés explicitement (6 matières physiques non-délégables)

**Tâches PO (non-délégables à Claude Code)** :
- Captures d'écran UI Jeedom (pas d'accès X11 fiable côté Claude Code)
- Validation Claude Desktop finale (UI sur machine perso PO)
- Sanity check sanitisation (œil humain qui connaît son install)
- Soumission market (compte développeur PO)
- Réponses forum communautaire (identité PO)
- Décisions / orientation / arbitrages

**Tâches Claude Code (toutes les autres)** : code, rédaction, tests, déploiements SSH, fixtures, logs, doc, releases.

Détail complet : `docs/state/CONTRIBUTING-CLAUDE-CODE.md`.

## Conséquences pressenties

- Le PO n'a pas besoin de lire le code pour valider l'avancement (résumés de session + PROJECT_STATE.md)
- Claude Code produit des ADRs + drafts que le PO critique, pas l'inverse
- Les sessions longues (marathon) sont à éviter : point de validation PO à chaque jalon ou lot de livrables
