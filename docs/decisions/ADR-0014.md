# ADR-0014 — Distribution market et politique de versioning semver

- **Date** : 2026-05-03 (esquisse) / 2026-05-03 (finalisation J0-1)
- **Statut** : proposed
- **Source brief** : D5.5, §12 (Distribution et maintenance), D10.3, D12.1-D12.9

## Contexte

Holmes MCP doit être distribué via le Market Jeedom officiel pour atteindre sa cible (Jeedomistes non-techniques). Le versioning doit être clair pour les utilisateurs, pour la skill jeedom-audit future consommatrice MCP (`holmesMcp >= 1.0, < 2.0`), et pour la maintenance.

## Décision — J0-1 (D10.3 + D12.6 arrêtés)

### D10.3 — docs/ embarquée dans `main` ✅

`docs/` reste dans la branche `main` (pas de branche `develop` dédiée à la doc). Rationale : le Market Jeedom zippe le plugin depuis la racine du repo — `docs/` est un dossier texte léger (quelques Mo max), aucun impact fonctionnel sur l'install. La simplicité d'un repo mono-branche pour la doc l'emporte sur l'avantage marginal d'une branche séparée.

### D12.6 — Documentation MkDocs Material ✅

Stack doc retenue : **MkDocs Material** hébergé sur GitHub Pages, source dans `docs/user/`, build automatisé via `.github/workflows/docs.yml` (déclenchement sur push `main`). Navigation, dark mode, recherche, sélecteur de version activés. URL : `https://isilorn.github.io/holmesMcp/`.

### Distribution et versioning

**Distribution** : Market Jeedom (voie nominale). Releases GitHub maintenues comme source canonique pour les power users.

**Versioning semver strict** :
- `v0.x.0` : pre-releases jalons internes J0-J7
- `v1.0.0` : première release stable (sortable quand 9 critères D8.3 validés)
- `v1.0.x` : patches (corrections, perfs, descriptions — schémas tools/resources **inchangés**)
- `v1.x.0` : minor (nouveaux tools/resources, paramètres optionnels rétro-compatibles)
- `v2.0.0` : major (breaking changes, migration guide obligatoire)

**Stratégie de sortie V1** :
- V1.0.0 → statut **bêta** sur le market (après bêta privée 2+ semaines sur box PO — D8.4)
- Conversion bêta → stable après quelques semaines de retours communautaires positifs (décision PO)

**Politique de support** :
- Jeedom N et N-1 supportés officiellement (alignement jeedom-skills)
- Spec MCP : N et N-1 supportés
- Breaking changes spec MCP → version majeure Holmes MCP

**Maintenance** :
- Bugs critiques (fuite, crash, faille) : patch sous 48h, annonce forum
- Bugs majeurs (tool données incorrectes, install échouant) : patch dans la semaine
- Bugs mineurs : prochaine release minor planifiée

**Procédure de release** : automatisée par Claude Code (tests CI verts → tests live via SSH → changelog + version → commit + tag → build zip → soumission market par PO ou API si disponible).

## Conséquences pressenties

- Les schémas des tools et resources sont contractuels dès V1.0.0 (jeedom-audit peut se binder dessus)
- Chaque release GitHub a des notes générées depuis `changelog.md`
- La soumission market est un goulet PO (compte développeur), la préparation est automatisée
