# ADR-0018 — Résultat POC D2.3 : faisabilité daemon Python sur Bookworm

- **Date** : 2026-05-03 (J0-2 + J0-3)
- **Statut** : accepted
- **Source brief** : D2.3 (POC faisabilité daemon Python sur Bookworm — 7 hypothèses)

## Contexte

Le brief identifiait 7 hypothèses à valider avant de s'engager sur l'architecture daemon Python + Streamable HTTP :

1. Python 3.11+ disponible sur Bookworm sans action manuelle
2. SDK MCP installable dans un venv Jeedom natif
3. FastMCP démarre et écoute sur le port configuré
4. `tools/list` retourne les tools enregistrés via le protocole MCP
5. Le daemon survive au redémarrage Jeedom (lancé/arrêté via `deamon_start/stop`)
6. PID file correct, `deamon_info()` retourne `state: ok`
7. Un client MCP peut se connecter au daemon depuis le LAN

L'hypothèse #7 était identifiée comme le principal risque (🔴 Élevé). Le brief mentionnait Claude Desktop comme client de validation. Le plan B prévu était HTTPS self-signed.

## Résultat des hypothèses (J0-2 + J0-3 — 2026-05-03)

| # | Hypothèse | Résultat |
| --- | --- | --- |
| 1 | Python 3.11+ sur Bookworm | ✅ Python 3.11.2 présent |
| 2 | SDK MCP installable en venv natif Jeedom | ✅ mcp==1.27.0 installé via packages.json |
| 3 | FastMCP démarre sur le port configuré | ✅ Uvicorn sur 0.0.0.0:8765 |
| 4 | `tools/list` retourne tool `hello` | ✅ Validé via curl depuis devbox |
| 5 | Daemon lancé/arrêté via deamon_start/stop | ✅ Fonctionne via PHP CLI Jeedom |
| 6 | PID file correct, state: ok | ✅ PID confirmé par deamon_info() |
| 7 | Client MCP connecté depuis le LAN | ✅ Claude Code HTTP natif validé en live |

**Hypothèse #7 : validée via Claude Code.** Le test initial avec Claude Desktop avait échoué non pas parce que HTTP est refusé, mais parce que Claude Desktop ne supporte pas le champ `url` dans `claude_desktop_config.json` — il n'accepte que `command`/`args` (stdio). Claude Code supporte HTTP LAN nativement (`type: "http"`).

## Décision

**Le daemon reste en HTTP. Pas de TLS, pas de certificat.**

Le plan B HTTPS self-signed est abandonné. L'architecture retenue pour les clients MCP V1 est la suivante :

### Clients supportés V1 (HTTP LAN natif)

| Client | Support HTTP LAN | Config requise |
| --- | --- | --- |
| **Claude Code** | ✅ Natif | `type: "http"`, `url`, `headers` dans `.mcp.json` |
| **Cursor** | ✅ Natif | Même format que Claude Code |
| **MCP Inspector** | ✅ Natif | URL directe dans l'UI |

**Les tests V1 sont réalisés en priorité dans Claude Code**, ce qui permet à Claude Code lui-même d'exécuter les tests via SSH sans intervention du PO (hypothèse #7 incluse).

### Claude Desktop : supporté via `mcp-remote`

Claude Desktop ne supporte pas les serveurs HTTP distants en LAN via son fichier de config. Il est néanmoins utilisable en installant `mcp-remote`, un pont stdio→HTTP maintenu par la communauté MCP :

```json
{
  "mcpServers": {
    "holmesMcp": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "http://<ip-jeedom>:8765/mcp",
        "--allow-http",
        "--header", "Authorization:${HOLMES_TOKEN}"
      ],
      "env": {
        "HOLMES_TOKEN": "Bearer <token>"
      }
    }
  }
}
```

Prérequis : Node.js + npx sur la machine du client. Cette procédure sera documentée dans la doc utilisateur MkDocs (section "Clients MCP").

### Clients desktop grand public : candidats V2+

La validation avec les clients desktop grand public (Claude Desktop natif si support HTTP LAN ajouté, ChatGPT si support MCP, autres) est repoussée en V2+. Ces clients évoluent rapidement et leur support HTTP LAN direct est susceptible d'arriver sans nécessiter de changement côté Holmes MCP.

## Conséquences

- Le daemon reste HTTP — aucun changement sur `holmesMcpd.py`, `post_install.sh`, `getMcpUrl()`
- Les tests d'intégration V1 (J3 à J6) sont exécutés par Claude Code via SSH sur la box PO
- La doc utilisateur MkDocs inclut une section "Claude Desktop via mcp-remote"
- ROADMAP.md : ajouter candidate V2+ "Validation clients desktop grand public"
- Le critère de sortie V1.0.0 D8.3 #4 ("Auth Bearer fonctionnel sur Claude Desktop ET au moins un autre client MCP") est amendé : **Claude Code + MCP Inspector** remplacent Claude Desktop comme clients de validation V1
