# ADR-0017 — Sanitisation et guardrails

- **Date** : 2026-05-03 (esquisse) / 2026-05-04 (proposed — J2-1 implémenté)
- **Statut** : proposed
- **Source brief** : §15 (Sanitisation et guardrails), D15.1-D15.10

## Contexte

La base Jeedom contient des données sensibles : tokens API plugins (Aqara, Tuya, Hue…), creds MQTT/broker, apikeys, mots de passe utilisateurs, configurations JSON imprévisibles dans les colonnes `configuration` des tables `eqLogic` et `cmd`. Sans filtrage, ces données partent vers les LLMs externes lors d'une requête innocente. La sanitisation est un **trait identitaire** de Holmes MCP, au même titre que la lecture seule V1.

## Décision

**Trois mécanismes cumulés** (défense en profondeur) — implémentés dans `_domain/sanitize.py` :

### Mécanisme 1 — Whitelist par table (D15.1)

13 tables whitélistées : `eqLogic`, `cmd`, `scenario`, `object`, `dataStore`, `config`, `plugin`, `history`, `historyArch`, `scenarioElement`, `scenarioSubElement`, `scenarioExpression`, `interactDef`. Pour chaque table, seuls les champs listés dans la whitelist sont exposés — tout champ hors whitelist est masqué par défaut.

### Mécanisme 2 — Regex sur clés JSON des blobs (D15.1)

Clés JSON des colonnes `configuration` / `options` / `value` filtrées par la regex :

```text
(?i)(password|pwd|token|apikey|api_key|secret|hash|credentials|auth|cert|private_key|access_key|client_secret|bearer)
```

Appliqué de façon récursive sur les blobs JSON imbriqués. Parsing resilient : si le JSON est invalide, la valeur brute est masquée entièrement.

### Mécanisme 3 — Liste hard-codée par plugin (D15.2)

15 plugins couverts (nommages exotiques non couverts par la regex) — liste V1 validée par le PO en J2-1 :

| Plugin | Champs hard-codés |
| --- | --- |
| jMQTT | `mqttUser`, `mqttPass`, `mqttApikey` |
| Aqara | `centralId`, `centralKey`, `apiKey` |
| Zigbee2MQTT | `mqttPass`, `apiKey` |
| Sonos | — (regex suffisante) |
| Philips Hue | `apiKey`, `bridgeApiKey` |
| Z-Wave | `apiKey` |
| Netatmo | `clientSecret`, `accessToken`, `refreshToken` |
| ecodevices | `apiKey` |
| rfxcom | — (regex suffisante) |
| Agenda | — (regex suffisante) |
| Alarme | `codeSecret`, `codeAlarm` |
| Jeedom Connect | `apiKey`, `deviceKey` |
| MQTT Manager | `mqttPass`, `mqttUser` |
| Script | — (regex suffisante) |
| Virtuel | — (regex suffisante) |

**4 plugins à vérifier live en J6 (ADR-0021)** : jMQTT, Alarme, Jeedom Connect, MQTT Manager — leurs champs exotiques seront validés sur la box PO avant V1.0.0.

### Comportement commun : Mask + count

- Champ filtré → remplacé par `"***FILTERED***"`
- Chaque réponse inclut `_filtered_fields` : liste des champs masqués (transparence LLM)
- Cas spécial `config` table : clé visible, valeur masquée si clé sensible (le LLM sait quel paramètre est configuré, pas sa valeur)

### Sanitisation `query_sql` (D15.3)

Refus HTTP 400 si la requête cible : tables `user`, `session`, `network` — ou colonnes `password` / `token` / `apikey` / `secret` / `hash` / `credentials`. Implémenté dans le tool `query_sql` (J5).

### Sanitisation des logs

Même sanitiseur appliqué aux logs JSON Lines du daemon. Aucun credential ne transite dans `holmesMcp_daemon.log`, même partiellement.

### Sanity check PO (D15.6)

Obligatoire avant V1.0.0 : Claude Code exécute les tools les plus exposants sur la box PO via SSH → rapport de sortie sanitisée fourni au PO → validation à l'œil humain. Non négociable. Planifié en J6.

### Procédure d'incident

En cas de fuite signalée : patch sous 24h, issue GitHub publique, annonce forum, ADR d'incident.

## Implémentation

- `_domain/sanitize.py` — J2-1 livré. 158 tests unitaires, **100% couverture** (D15.5 ✅).
- `_domain/cmd_refs.py` — J2-3 livré. Résolveur `#cmdId#` → `#[O][E][C]#` utilisé dans `describe_scenario` (J3).

## Conséquences

- Un défaut dans un tool ne peut pas exposer de credentials : 3 mécanismes doivent échouer simultanément
- La communauté peut enrichir la liste hard-codée via issues GitHub (template "Champ sensible non filtré")
- La transparence LLM-side (mask + count) évite les hallucinations sur des champs silencieusement supprimés
