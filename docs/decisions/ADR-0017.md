# ADR-0017 — Sanitisation et guardrails

- **Date** : 2026-05-03 (esquisse) / TBD (finalisation J0)
- **Statut** : draft
- **Source brief** : §15 (Sanitisation et guardrails), D15.1-D15.10

## Contexte

La base Jeedom contient des données sensibles : tokens API plugins (Aqara, Tuya, Hue…), creds MQTT/broker, apikeys, mots de passe utilisateurs, configurations JSON imprévisibles dans les colonnes `configuration` des tables `eqLogic` et `cmd`. Sans filtrage, ces données partent vers les LLMs externes lors d'une requête innocente. La sanitisation est un **trait identitaire** de Holmes MCP, au même titre que la lecture seule V1.

## Décision pressentie

**Trois mécanismes cumulés** (défense en profondeur) :

1. **Whitelist de champs exposables** sur tables connues (`eqLogic`, `cmd`, `scenario`, etc.). Tout champ hors whitelist est masqué par défaut.

2. **Regex sur clés JSON** des blobs `configuration`/`options`/`value` :
   `(?i)(password|pwd|token|apikey|api_key|secret|hash|credentials|auth|cert|private_key|access_key|client_secret|bearer)`

3. **Liste hard-codée par plugin connu** (nommages exotiques non couverts par la regex) — liste V1 arrêtée en J1-1 (D15.2 ✅) : jMQTT, Aqara, Zigbee2MQTT, Sonos, Philips Hue, Z-Wave, Netatmo, ecodevices, rfxcom, agenda (10 plugins les plus installés). Implémentation complète en J2 (`_domain/sanitize.py`).

**Comportement** : **Mask + count** — champs remplacés par `***FILTERED***` + tableau `_filtered_fields` dans chaque réponse. Le LLM sait qu'un champ est masqué et peut l'expliquer à l'utilisateur.

**Sanitisation des logs** : même sanitiseur que les réponses MCP. Pas de credential dans les logs même partiellement.

**`query_sql` restrictions spécifiques** (D15.3) : refus 400 si la requête cible la table `user`, `session`, `network`, ou des colonnes nommées `password`/`token`/`apikey`/`secret`/`hash`/`credentials`.

**Sanity check PO obligatoire** avant V1.0.0 (D15.6) : Claude Code exécute les tools les plus exposants sur la box réelle via SSH, fournit un rapport de sortie sanitisée au PO, qui valide à l'œil humain. Non négociable.

**Procédure de réponse rapide** en cas de fuite signalée : patch sous 24h, issue GitHub publique, annonce forum, ADR d'incident.

**Tests** (D15.5) : couverture 100% obligatoire sur `_domain/sanitize.py`. Fixtures avec credentials connus dans tous les emplacements — vérification que la sortie ne contient aucune occurrence.

## Conséquences pressenties

- Un défaut dans un tool ne peut pas exposer de credentials : 3 mécanismes doivent échouer simultanément
- La communauté peut enrichir la liste hard-codée via issues GitHub (template "Champ sensible non filtré")
- La transparence LLM-side (mask + count) évite les hallucinations du LLM sur des champs silencieusement supprimés
