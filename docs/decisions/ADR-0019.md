# ADR-0019 — Couverture fonctionnelle skill jeedom-audit (D5.8)

- **Date** : 2026-05-03 (J1-2)
- **Statut** : accepted
- **Source brief** : D5.8 (matrice couverture WF1-WF13 skill jeedom-audit ↔ tools Holmes MCP V1)

## Contexte

Le brief Holmes MCP prévoit qu'à terme, la skill jeedom-audit basculera en consommatrice exclusive de Holmes MCP pour ses accès aux données Jeedom (D5.8). Avant de s'engager sur ce couplage, il faut vérifier que le périmètre V1 des 25 tools MCP couvre bien tous les workflows actuels WF1–WF13 de jeedom-audit, sans perte de capacité.

**Référence jeedom-audit analysée :** commit `a792179`, tag `v1.0.1`, branche `main` (repo `jeedom-skills`).

## Décision

**Les 25 tools Holmes MCP V1 couvrent intégralement les 13 workflows WF1–WF13 de jeedom-audit. Aucun ajout au périmètre D5.3 n'est requis.**

Deux points de simplification notable par rapport à jeedom-audit :

1. **`router.py` non porté** : inutile. Le daemon Holmes MCP tourne sur la box — MySQL et API JSON-RPC sont toujours disponibles simultanément. La notion de "mode API-only" de jeedom-audit (sans SSH) disparaît.

2. **`resolve_cmd_refs.py` non porté comme module séparé** : la résolution `#ID#` → `#[O][E][C]#` est intégrée directement dans `describe_scenario`. Pour les besoins ponctuels (SQL ad-hoc), `find_commands_advanced` par `cmd_id` suffit.

## Conséquences

- La liste D5.3 (25 tools) est **figée pour V1** — pas d'ajout issu de D5.8.
- La matrice `docs/skill-coverage-matrix.md` est la référence vivante de ce couplage. Mettre à jour si jeedom-audit évolue.
- La bascule jeedom-audit → consommatrice Holmes MCP reste candidate roadmap post-V1. Elle sera réalisable sans modification du périmètre tools.
- D5.8 dans `PROJECT_STATE.md` : tranché.
