# ADR-0005 — Canaux d'accès aux données Jeedom

- **Date** : 2026-05-03 (esquisse) / TBD (finalisation J0)
- **Statut** : draft
- **Source brief** : §4bis (Accès aux données Jeedom), D4bis.1-D4bis.7

## Contexte

Le daemon Holmes MCP tourne sur la box, ce qui lui donne accès local direct aux 3 canaux de données Jeedom : MySQL, fichiers de log, API JSON-RPC. Contrairement à jeedom-audit (qui passe par SSH depuis l'extérieur), Holmes MCP peut accéder directement à ces ressources. Le canal MySQL doit être garanti read-only à perpétuité, quel que soit l'état du plugin (même en cas de bug ou de prompt injection).

## Décision pressentie

**Trois canaux d'accès distincts**, routage hard-codé dans le daemon (D4bis.6) :

| Opération | Canal | Implémentation |
|---|---|---|
| Audit structurel | MySQL (user `jeedom_mcp_ro` — SELECT uniquement) | driver Python sur localhost:3306 |
| État runtime | API JSON-RPC localhost | HTTP POST `http://127.0.0.1/core/api/jeeApi.php` |
| Historique récent | API JSON-RPC | idem |
| Historique archivé | MySQL (`historyArch`) | idem MySQL |
| Logs | Lecture fichier local | `_core/logs.py` dérivé de `logs_query.py` |
| Écriture V2+ | API JSON-RPC **uniquement** | **Jamais via SQL** |

**Garantie technique read-only MySQL** : le user `jeedom_mcp_ro` n'a que `GRANT SELECT ON jeedom.* TO 'jeedom_mcp_ro'@'localhost'` — aucune requête modifiante possible côté daemon, même en cas de bug. Ce GRANT ne sera jamais étendu en V1 ni dans les versions futures pour ce canal.

**Création / suppression du user MySQL** aux hooks install/remove du plugin. Password aléatoire stocké via `config::save()`. Passé au daemon en CLI arg au démarrage (pattern template Jeedom).

**Driver MySQL** : PyMySQL recommandé (D4bis.1, décision 🔵 à confirmer J0). Critères : pure Python, maintenance active, compatible MySQL 5.7 + MariaDB 10.x.

**Blacklist V1 des méthodes API JSON-RPC modifiantes** (D4bis.5) : `cmd::execCmd`, `scenario::changeState`, `datastore::save`, `interact::tryToReply`, et tout pattern dont le nom suggère une modification. En V2+, bascule vers whitelist explicite (approche positive).

## Conséquences pressenties

- Garantie formelle : un défaut dans n'importe quel tool Holmes MCP ne peut pas écrire dans MySQL
- L'apikey JSON-RPC globale Jeedom passe au daemon via CLI arg au démarrage (pattern sécurisé template Jeedom) — jamais dans le code
- Les credentials MySQL (host, user, password RO) ne transitent pas par le repo (D11.8)
