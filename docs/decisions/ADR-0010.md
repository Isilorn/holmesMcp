# ADR-0010 — Nom et identité produit Holmes MCP

- **Date** : 2026-05-03 (esquisse) / TBD (finalisation J0)
- **Statut** : draft
- **Source brief** : D10.8

## Contexte

Le plugin doit avoir un ID technique Jeedom (camelCase, unique sur le market) et un nom human-readable cohérent avec son positionnement produit. L'identité narrative doit refléter les traits distinctifs : observateur méthodique, lecture seule, sanitisation forte, pas d'action sur l'installation.

## Décision pressentie

**ID Jeedom (technique)** : `holmesMcp`
**Nom human-readable** : "Holmes MCP"

**Identité produit** :
> *Holmes MCP : Holmes observe, déduit, raconte. Il ne touche jamais à la scène, sans jamais exposer les secrets de votre maison. Plugin Jeedom natif open source qui expose la box Jeedom comme serveur MCP en lecture seule, orienté audit, diagnostic et assistance à la conception. Pour Claude Desktop, Cursor et tout client MCP compatible.*

La vérification que `holmesMcp` est libre comme ID Jeedom market et l'absence de collision marque évidente sont **déléguées à Claude Code J0** (🔵 D10.8). Si conflit identifié → escalade PO avec alternatives (`jeedoscope`, `vigieMcp`, `holmesMcpAudit`, autre).

Cette identité est à mentionner dans : le README market (racine), la doc utilisateur MkDocs (section 1), et ADR-0010 elle-même.

## Conséquences pressenties

- L'ID `holmesMcp` est utilisé partout dans la structure du plugin (nommage fichiers PHP, config Jeedom, logs)
- La narrative "observe/déduit/raconte/sans toucher" doit rester cohérente avec les évolutions V2+ (l'écriture V2+ ne casse pas l'identité — Holmes peut aussi agir sur demande explicite, il reste méthodique)
- L'icon market doit refléter l'identité (Holmes-themed, à concevoir et soumettre au PO J7)
