# ADR-0021 — Stratégie d'adressage et de test par plugin (install PO)

- **Date** : 2026-05-04 (J2-1), révisé J2-1 post-discussion
- **Statut** : accepted
- **Source brief** : D15.2 (plugins hard-codés sanitiseur), D5.6 (query_sql), D15.5-D15.6 (tests + sanity check), ADR-0018 (stratégie tests SSH)

## Contexte

Deux questions émergent après la liste de plugins fournie par le PO en J2-1 :

1. Certains plugins créent des tables MySQL custom non couvertes par les tools génériques.
2. Les extras sanitiseur de mech 3 (`_PLUGIN_EXTRA_KEYS`) nomment des clés supposées,
   non vérifiées sur la box réelle.

**Principe acté** : la box PO est une ressource de test pour **tous** les plugins
installés, pas seulement pour le sanity check final J6. La stratégie de test par
niveau est :

| Niveau | Moyen | Portée | Jalon |
|---|---|---|---|
| **Tests unitaires** | Fixtures synthétiques | Algorithme pur (sanitiseur, walker, graph) | J2+ |
| **Tests d'intégration** | Box PO via SSH (ADR-0018) | Comportement réel — tous les plugins installés | J3-J4 |
| **Sanity check** | PO à l'œil | Lecture humaine du rapport sanitisé complet | J6 (D15.6) |

Les tests d'intégration SSH sur box (déjà établis en J1-3) s'appliquent
**à tous les plugins installés** dès J3-J4 — structure réelle de configuration,
tables effectivement présentes, données non mockées.

---

## Périmètre

**8 plugins de l'install PO** : jMQTT, Agenda, Alarme, Thermostat, Jeedom Connect,
Script, MQTT Manager, Virtuel.

---

## Analyse par plugin

### jMQTT

**Rôle** : équipements MQTT — broker + topics/commandes.

**Tables** :
- Standard : `eqLogic` (1 eqLogic = broker), `cmd` (1 cmd = 1 topic abonné/publié)
- Custom possible : `jmqtt_cmd` (MQTT topic par commande, versions antérieures).
  En jMQTT v4+, le topic est stocké dans `cmd.logicalId` — table custom absente ou vide.

**Tool access** :
- `get_equipment` + `list_commands` : broker + topics (suffisant pour l'audit)
- `query_sql` : `jmqtt_cmd` si présente (cookbook V1)

**Sanitiseur — extras supposés** :
```
configuration JSON du broker eqLogic :
  mqttpassword / mqtt_password  ← regex mech 2 couvre
  mqttuser / mqtt_user / broker_user / login / username  ← mech 3 (à confirmer)
```

**Tests** :
- Unitaires : fixture synthétique avec blob configuration jMQTT
- Intégration J3-J4 : `get_equipment` live sur box → vérifie structure réelle +
  confirme que les extras sanitiseur couvrent les vraies clés + détecte jmqtt_cmd

---

### Agenda

**Rôle** : équipements calendrier / événements planifiés.

**Tables** :
- Standard : `eqLogic`, `cmd`
- Custom : `calendar_event` (événements : heure, récurrence, commandes déclenchées)

**Tool access** :
- `get_equipment` + `list_commands` : équipement calendrier
- `query_sql` : `calendar_event` pour les événements planifiés (cookbook V1)

**Sanitiseur** : aucun credential. Pas d'extras plugin requis.

**Tests** :
- Unitaires : fixture synthétique standard
- Intégration J3-J4 : `get_equipment` live + `query_sql SELECT * FROM calendar_event LIMIT 5`
  → confirme structure table custom

---

### Alarme

**Rôle** : système d'alarme — zones, modes, codes d'armement.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands` (modes, zones, états).

**Sanitiseur — extras supposés** :
```
configuration JSON eqLogic Alarme :
  armCode / disarmCode / pin / code / userCode  ← mech 3 (à confirmer)
```

**Tests** :
- Unitaires : fixture synthétique avec codes armement dans configuration
- Intégration J3-J4 : `get_equipment` live sur box → confirme nommage réel des codes +
  vérifie que la sortie sanitisée ne contient aucun code

---

### Thermostat

**Rôle** : régulateur de température — consigne, mode, commandes liées.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands`.

**Sanitiseur** : aucun credential. Configuration référence des commandes via `#cmdId#`.

**Tests** :
- Unitaires : fixture synthétique standard
- Intégration J3-J4 : `get_equipment` live → vérifie structure configuration (liens `#cmdId#`)

---

### Jeedom Connect

**Rôle** : application mobile — chaque appareil = 1 eqLogic avec token de jumelage.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands`.

**Sanitiseur — extras supposés** :
```
configuration JSON eqLogic appareil :
  installCode / access_code / pairing_code / device_code  ← mech 3 (à confirmer)
  token / api_key / secret  ← regex mech 2 couvre si présents
```

**Tests** :
- Unitaires : fixture synthétique avec token dans configuration
- Intégration J3-J4 : `get_equipment` live → confirme nommage réel des tokens +
  vérifie masquage dans la sortie

---

### Script

**Rôle** : équipements scripting — bash, python, php, ruby.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands`.

**Sanitiseur** : pas de credentials structurels propres au plugin.
Risque résiduel : credentials embarqués dans un script non détectés (hors périmètre V1).

**Tests** :
- Unitaires : fixture synthétique standard
- Intégration J3-J4 : `get_equipment` live → vérifie structure (chemins scripts, types)

---

### MQTT Manager

**Rôle** : broker MQTT intégré — gestion des clients, topics, credentials.

**Tables** :
- Standard : `eqLogic`, `cmd`
- Custom possible : tables propres au broker embarqué (non confirmé)

**Tool access** :
- `get_equipment` + `list_commands` : configuration broker
- `query_sql` : à confirmer si tables custom présentes

**Sanitiseur — extras supposés** :
```
configuration JSON eqLogic MQTT Manager :
  broker_user / broker_login / login / user / username  ← mech 3 (à confirmer)
  password / broker_password  ← regex mech 2 couvre
```

**Tests** :
- Unitaires : fixture synthétique avec credentials broker dans configuration
- Intégration J3-J4 : `get_equipment` live + `SHOW TABLES LIKE 'mqtt%'` →
  confirme nommage réel + détecte tables custom éventuelles

---

### Virtuel

**Rôle** : équipements virtuels — agrège ou calcule des valeurs depuis d'autres commandes.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands`.

**Sanitiseur** : aucun credential structurel. Configuration référence des commandes via `#cmdId#`.

**Tests** :
- Unitaires : fixture synthétique standard
- Intégration J3-J4 : `get_equipment` live → vérifie formules et références

## Décision — Matrice récapitulative

| Plugin | Tables custom | query_sql cookbook | Extras sanitiseur | Tests unitaires | Tests intégration (J3-J4) |
|---|---|---|---|---|---|
| jMQTT | jmqtt_cmd (possible) | ✅ jmqtt_cmd | mqttuser, login… | Fixture | Box SSH — structure + sanitiseur |
| Agenda | calendar_event | ✅ calendar_event | aucun | Fixture | Box SSH — structure + calendar_event |
| Alarme | Non | Non | armCode, pin… | Fixture | Box SSH — structure + sanitiseur |
| Thermostat | Non | Non | aucun | Fixture | Box SSH — structure |
| Jeedom Connect | Non | Non | installCode… | Fixture | Box SSH — structure + sanitiseur |
| Script | Non | Non | aucun | Fixture | Box SSH — structure |
| MQTT Manager | Possible | À confirmer | broker_user, login… | Fixture | Box SSH — structure + sanitiseur + tables |
| Virtuel | Non | Non | aucun | Fixture | Box SSH — structure |

**Colonne "Tests intégration (J3-J4)"** : chaque plugin installé fait l'objet d'un test
d'intégration SSH sur box lors de l'implémentation du tool qui le couvre. Pour les plugins
avec extras sanitiseur supposés, ce test vérifie aussi que les vrais noms de champs sont
bien couverts — et corrige `_PLUGIN_EXTRA_KEYS` si nécessaire (patch avant J6, pas après).

## Conséquences

### Sur J2-2 (usage_graph + scenario_walker)

Pas d'impact. Les fixtures J2-2 modélisent des structures de scénarios, pas des
configurations plugin.

### Sur J3-J4 (18 tools)

- Tests d'intégration SSH sur box pour les 8 plugins installés, intégrés au flux normal
  (même pattern que J1-3 pour `_core/`).
- Si un extras sanitiseur s'avère faux ou incomplet → correction `_PLUGIN_EXTRA_KEYS`
  en J3-J4, pas en J6. J6 n'est pas le premier filet, c'est le dernier.
- Le `query_sql` SQL cookbook documente les 2 tables custom connues : `jmqtt_cmd`,
  `calendar_event`. MQTT Manager : à compléter après inspection J3-J4.

### Sur J6 (D15.6 — sanity check PO)

Rôle recentré : **lecture humaine du rapport sanitisé complet** par le PO sur sa box.
Les vérifications techniques (noms de champs, tables) ont déjà été faites en J3-J4.
J6 reste non négociable — le PO connaît son install mieux que les tests automatisés.
