# ADR-0021 — Stratégie d'adressage et de test par plugin (install PO)

- **Date** : 2026-05-04 (J2-1)
- **Statut** : accepted
- **Source brief** : D15.2 (plugins hard-codés sanitiseur), D5.6 (query_sql), D15.5-D15.6 (tests + sanity check)

## Contexte

La liste de plugins hard-codés dans `_domain/sanitize.py` (J2-1) est produite sans
vérification live des noms de champs réels dans les configurations Jeedom de l'install PO.
Par ailleurs, certains plugins créent des tables MySQL custom non couvertes par les tools
génériques (get_equipment, list_commands…). Sans document de référence, les risques sont :

1. **Sanitiseur borgne** : les extras plugin de mech 3 (`_PLUGIN_EXTRA_KEYS`) nomment
   des clés supposées — si le plugin utilise un nommage différent, le credential passe.
2. **Tool manquant** : un plugin avec une table custom utile au diagnostic n'est pas
   adressable autrement que par `query_sql`, sans SQL cookbook documenté.
3. **Fixtures inadaptées** : les fixtures synthétiques J2-2/J3-J4 ne reflètent pas la
   réalité si la structure de configuration d'un plugin est inconnue.

Déclencheur : PO a fourni sa liste de 8 plugins en J2-1. Décision de clarifier la
stratégie plugin avant J2-2.

---

## Périmètre

**8 plugins de l'install PO** : jMQTT, Agenda, Alarme, Thermostat, Jeedom Connect,
Script, MQTT Manager, Virtuel.

---

## Analyse par plugin

### jMQTT

**Rôle** : équipements MQTT — broker + topics/commandes.

**Tables** :
- Standard : `eqLogic` (1 eqLogic = broker), `cmd` (1 cmd = 1 topic abonné/publié)
- Custom possible : `jmqtt_cmd` (MQTT topic par commande, versions antérieures).
  En jMQTT v4+, le topic est stocké dans `cmd.logicalId` — table custom absente ou vide.

**Tool access** :
- `get_equipment` + `list_commands` : broker + topics (suffisant pour l'audit)
- `query_sql` : `jmqtt_cmd` si présente (cookbook V1)

**Sanitiseur — champs supposés** (non vérifiés sur box PO) :
```
configuration JSON du broker eqLogic :
  mqttpassword / mqtt_password / broker_password  ← regex mech 2 couvre
  mqttuser / mqtt_user / broker_user / login / username  ← mech 3 couvre (supposé)
```
**⚠️ Vérification box requise (J6)** : confirmer les noms exacts des clés de
configuration du broker jMQTT sur la box PO.

---

### Agenda

**Rôle** : équipements calendrier / événements planifiés.

**Tables** :
- Standard : `eqLogic`, `cmd`
- Custom : `calendar_event` (événements : heure, récurrence, commandes déclenchées).
  Table connue de jeedom-audit (`calendar_event.repeat` — cf. db_query.py commentaire).

**Tool access** :
- `get_equipment` + `list_commands` : équipement calendrier (suffisant pour la plupart)
- `query_sql` : `calendar_event` pour consulter les événements planifiés (SQL cookbook V1)

**Sanitiseur** : aucun credential dans configuration Agenda. Pas d'extras plugin requis.

**✅ Fixture suffisante** : structure standard connue, pas de credential.

---

### Alarme

**Rôle** : système d'alarme — zones, modes, codes d'armement.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands` (modes, zones, états).

**Sanitiseur — champs supposés** (non vérifiés sur box PO) :
```
configuration JSON de l'eqLogic Alarme :
  armCode / disarmCode / pin / code / userCode  ← mech 3 (supposé)
```
**⚠️ Vérification box requise (J6)** : le plugin Alarme peut nommer ses codes
différemment selon la version (`user_code`, `alarm_pin`, etc.). Confirmer sur box PO.

---

### Thermostat

**Rôle** : régulateur de température — consigne, mode, commandes liées.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands` (consigne, mode, capteur lié via logicalId).

**Sanitiseur** : aucun credential. La configuration référence des commandes Jeedom
via `#cmdId#` — pas de credentials.

**✅ Fixture suffisante** : structure standard connue, pas de credential.

---

### Jeedom Connect

**Rôle** : application mobile — chaque appareil = 1 eqLogic avec token de jumelage.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands` (commandes remontées par l'app).

**Sanitiseur — champs supposés** (non vérifiés sur box PO) :
```
configuration JSON de l'eqLogic appareil :
  installCode / access_code / pairing_code / device_code  ← mech 3 (supposé)
  token / api_key / secret  ← regex mech 2 couvre si présents
```
**⚠️ Vérification box requise (J6)** : les noms exacts des champs de token dans
la configuration Jeedom Connect sont inconnus sans inspection live.

---

### Script

**Rôle** : équipements scripting — bash, python, php, ruby.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands` (script path, type, paramètres).

**Sanitiseur** : le chemin du script est dans `cmd.configuration` ou `cmd.logicalId`,
pas des credentials à proprement parler. Si un script embarque des credentials en
argument, ceux-ci seraient dans la configuration — couverts par mech 2 si les clés
sont nommées de façon standard.

**✅ Fixture suffisante** : pas de credentials structurels propres au plugin Script.
Risque résiduel : credentials embarqués dans un script non détectés (hors périmètre V1).

---

### MQTT Manager

**Rôle** : broker MQTT intégré — gestion des clients, topics, credentials.

**Tables** :
- Standard : `eqLogic`, `cmd`
- Custom possible : tables propres au broker embarqué (non confirmé).

**Tool access** :
- `get_equipment` + `list_commands` : configuration broker
- `query_sql` : à confirmer si tables custom présentes

**Sanitiseur — champs supposés** (non vérifiés sur box PO) :
```
configuration JSON eqLogic MQTT Manager :
  broker_user / broker_login / login / user / username  ← mech 3 (supposé)
  password / broker_password  ← regex mech 2 couvre
```
**⚠️ Vérification box requise (J6)** : confirmer noms de champs réels + vérifier
si MQTT Manager crée des tables custom.

---

### Virtuel

**Rôle** : équipements virtuels — agrège ou calcule des valeurs depuis d'autres commandes.

**Tables** : standard uniquement (`eqLogic`, `cmd`).

**Tool access** : `get_equipment` + `list_commands` (formule de calcul dans logicalId/configuration).

**Sanitiseur** : aucun credential structurel. La configuration référence des
commandes via `#cmdId#`.

**✅ Fixture suffisante** : structure standard connue, pas de credential.

## Conséquences

### Sur J2-2 (usage_graph + scenario_walker)

Les fixtures synthétiques J2-2 n'ont pas besoin de simuler des configurations plugin —
elles modélisent des structures de scénarios, pas des credentials. Pas de blocage.

### Sur J3-J4 (18 tools)

- `get_equipment` est générique et fonctionne pour tous les plugins via `eqLogic.configuration`
  (sanitisé par mech 2+3).
- Le `query_sql` SQL cookbook (description du tool, D5.6) doit inclure des exemples
  pour `jmqtt_cmd` et `calendar_event` — 2 tables custom documentées.

### Sur le sanitiseur (existant J2-1)

Les extras plugin de `_PLUGIN_EXTRA_KEYS` pour jMQTT, Alarme, Jeedom Connect et
MQTT Manager sont **provisoires** jusqu'à vérification box J6. Si J6 révèle des
champs manquants, un patch `v1.0.x` est émis (semver patch — schéma stable, ajout liste).

### Sur J6 (D15.6 — sanity check PO)

Checklist de vérification live obligatoire :

1. jMQTT : `SELECT configuration FROM eqLogic WHERE eqType_name='jMQTT' LIMIT 1`
   → vérifier clés broker credentials dans le JSON retourné
2. Alarme : `SELECT configuration FROM eqLogic WHERE eqType_name='Alarme' LIMIT 1`
   → vérifier clés codes armement/désarmement
3. Jeedom Connect : `SELECT configuration FROM eqLogic WHERE eqType_name='jeedomConnect' LIMIT 1`
   → vérifier clés tokens d'appareils
4. MQTT Manager : `SELECT configuration FROM eqLogic WHERE eqType_name='MQTT Manager' LIMIT 1`
   + `SHOW TABLES LIKE 'mqtt%'` → vérifier champs + existence tables custom

Si une clé réelle ne figure pas dans `_PLUGIN_EXTRA_KEYS`, elle est couverte si elle
matche la regex mech 2. Sinon → ajout dans mech 3, patch `v1.0.x`.
