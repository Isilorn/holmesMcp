# ADR-0004 — Authentification MCP externe

- **Date** : 2026-05-03 (esquisse) / TBD (finalisation J0)
- **Statut** : draft
- **Source brief** : §4 (Authentification MCP externe), D4.1-D4.7

## Contexte

Le daemon MCP est exposé sur le réseau LAN (voire Internet via reverse proxy). Tout client capable de joindre le port peut tenter une requête. Trois scopes d'auth coexistent : (1) MCP externe (client → daemon — sujet de cette ADR), (2) interne PHP↔daemon (mécanismes Jeedom natifs, hors scope), (3) daemon → API JSON-RPC Jeedom (apikey globale, hors scope). Le standard de fait MCP est le header `Authorization: Bearer <token>`.

Le daemon doit aussi attribuer chaque requête à un utilisateur Jeedom pour les logs (D4.7) et, à terme, pour les scopes par rôle (roadmap V1.x).

## Décision pressentie

**Token MCP dédié par utilisateur Jeedom**, stocké dans la table `config` Jeedom via `config::save('token_<user_id>', token, __CLASS__)` / `config::byKey('token_<user_id>', __CLASS__)`. Résolution token → login via `JOIN config ↔ user` (plugin='holmesMcp', key LIKE 'token_%', user.enable=1). Transmis en header `Authorization: Bearer <token>`.

- **À l'installation** : génération automatique d'un token UUID 32+ caractères (source aléatoire cryptographique) pour chaque utilisateur Jeedom existant
- **Page de config plugin** : tableau "Utilisateurs Jeedom et leurs tokens MCP" — nom, token (masqué par défaut), bouton "Régénérer" individuel, bouton "Révoquer" individuel
- **URL complète affichée** dans la page de config (form : `http://<ip-jeedom>:<port>/mcp`) avec le token — source de vérité zéro ambiguïté
- **401** sur toute requête sans token valide, pas de fallback lecture limitée
- **Logs attribués** à l'utilisateur Jeedom résolu depuis le token (champ `user` dans JSON Lines D14.1)

**V1 : scope unique** (tous les utilisateurs MCP ont les mêmes droits). Différenciation admin/user = roadmap V1.x. Multi-tokens par utilisateur = roadmap V1.x.

**HTTP par défaut V1.** TLS = responsabilité du reverse proxy utilisateur (nginx, traefik, Caddy, Cloudflare Tunnel). Si POC J0 révèle que Claude Desktop refuse HTTP non-TLS en LAN → plan B HTTPS self-signed activé sans hésitation (ADR à rédiger post-POC).

## Conséquences pressenties

- La page de config plugin est la source de vérité pour l'URL + token de connexion
- Tout utilisateur Jeedom avec un token peut invoquer tous les tools V1 (pas de granularité V1)
- Les logs permettent de tracer qui a invoqué quoi (audit, détection anomalie)
- Rotation individuelle des tokens disponible sans impact sur les autres utilisateurs
