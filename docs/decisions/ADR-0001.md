# ADR-0001 — Architecture Holmes MCP

- **Date** : 2026-05-03 (esquisse) / TBD (finalisation J0)
- **Statut** : draft
- **Source brief** : §2 (Architecture du plugin), §3 (Transport MCP), D2.1, D2.2, D3.1

## Contexte

Holmes MCP est un plugin Jeedom natif qui doit exposer la box comme serveur MCP accessible depuis un client distant (Claude Desktop, Cursor…). Trois options architecturales ont été évaluées en session d'idéation (ADR-0019 sources, superseded) : (A) MCP server local sur la machine cliente, (B) plugin PHP pur, (C) plugin PHP + daemon Python. L'option C a été retenue (ADR-0020 sources, accepted).

Le pattern "plugin PHP enveloppe + daemon Python lancé par les hooks Jeedom standards" est documenté officiellement par Jeedom et utilisé par des plugins majeurs (jMQTT, Mobile). Il permet de tirer parti du SDK MCP Python officiel d'Anthropic tout en intégrant le plugin dans l'écosystème Jeedom (market, gestion cycle de vie, UI config).

## Décision pressentie

**PHP fait uniquement l'enveloppe market** : manifeste `info.json`, hooks `deamon_start()` / `deamon_stop()` / `deamon_info()` / `dependancy_info()` / `dependancy_install()`, page de configuration UI (tokens par user, port d'écoute, niveau de log), callback `jeeholmesMcp.php`. PHP ne touche jamais au protocole MCP.

**Le daemon Python 3.11+** (processus permanent lancé par PHP) héberge l'intégralité du serveur MCP : transport Streamable HTTP (spec MCP 2025-03-26+), registration des 25 tools et 5 resources, auth Bearer, accès aux données Jeedom (MySQL, logs, API JSON-RPC), sanitisation.

## Conséquences pressenties

- Réutilisation forte des scripts jeedom-audit existants (D7.1) dans le daemon Python
- Diagnostic accessible : le PO peut voir l'état du daemon dans l'UI Jeedom standard (cadre "Démon" vert/rouge)
- Un POC (D2.3) est requis avant engagement de la suite pour valider les 7 hypothèses d'intégration
- PHP ne peut jamais être involontairement modifié pour exposer des capacités MCP (séparation stricte des responsabilités)
