## Architecture de `query_sql`

### Pipeline de validation (ordre strict)

```
sql → _check_select_only → _extract_table_names → _check_blacklist
    → _check_sensitive_columns → _ensure_limit → _db.query → sanitize_rows → wrap_result
```

### Fonctions internes

| Fonction | Rôle |
|---|---|
| `_check_select_only(sql)` | `sqlparse.get_type()` — refuse tout statement non-SELECT, refuse les statements multiples |
| `_extract_table_names(sql)` | Regex FROM/JOIN avec backticks optionnels — retourne liste lowercase |
| `_check_blacklist(tables)` | Frozenset `{user, session, network}` + regex `creds?/credentials?/password*/token*` |
| `_check_sensitive_columns(sql)` | D15.3 — regex sur le SELECT uniquement (pas WHERE), refuse `password/token/apikey/secret/private_key…` |
| `_ensure_limit(sql)` | Injecte `LIMIT 50` si absent ; plafonne à `200` si dépassé |

### Sanitisation résultats

- Requête mono-table connue → `sanitize_rows(rows, table=primary_table)` (whitelist active)
- Requête multi-table ou table inconnue → `sanitize_rows(rows, table=None)` (régex seule)
- Table `config` → `_sanitize_config_row` actif (key visible, value masquée si key sensible)

## Décisions prises en session

**`SELECT *` non bloqué par D15.3 — la sanitisation gère**  
`SELECT password FROM config` → refusé (D15.3). `SELECT * FROM config` → autorisé, mais
`sanitize_rows(rows, table='config')` masque automatiquement les valeurs sensibles via
`_sanitize_config_row`. Choix : `SELECT *` est le cas d'usage légitime courant du LLM ;
le refuser serait sur-restrictif. La sanitisation constitue la vraie défense en profondeur.

**D15.3 porte sur le SELECT uniquement (pas le WHERE)**  
`SELECT key, value FROM config WHERE key = 'password'` est autorisé — la clause WHERE
filtre les résultats mais ne demande pas à lire la colonne `password` elle-même. Le WHERE
est une condition de filtrage, pas une extraction. La valeur retournée passe par
`_sanitize_config_row` de toute façon.

**Famille 7 (pas family_sql ou family_query)**  
Cohérence de nommage avec les 6 familles existantes. `query_sql` est le seul tool de
cette famille dans V1 — le découplage en famille séparée reste aligné sur la structure
D5.3 du brief.

**LIMIT default=50, max=200**  
Default 50 : valeur suffisante pour la majorité des explorations LLM sans surcharger la
réponse. Max 200 : plafond raisonnable pour des requêtes analytiques légitimes sans risque
de timeout. Cohérent avec les limites des autres tools (list_equipments max=100,
list_commands max=200).

**Table primaire pour sanitisation = première table FROM**  
Pour les requêtes mono-table, la première table de la clause FROM est utilisée comme hint
pour `sanitize_rows`. Pour les jointures multi-tables, `table=None` est passé — la
whitelist n'est pas applicable car les colonnes peuvent venir de n'importe quelle table.
La regex (mécanisme 2) reste active dans les deux cas.
