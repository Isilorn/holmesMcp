## Décisions prises en session

**`get_health_summary` — MySQL uniquement (pas API JSON-RPC)**  
Les données de santé (daemon status, messages, crons) sont disponibles directement en MySQL
via les tables `update`, `message`, `cron`. L'API JSON-RPC n'est pas nécessaire pour ce tool.
Les champs datetime sont sérialisés avec `str(r['date'])` pour éviter les problèmes de
sérialisation JSON si PyMySQL retourne des objets `datetime.datetime`. Validé en J5-3 sur
box réelle.

**`search_text` — 4 requêtes séparées (pas UNION)**  
Quatre requêtes indépendantes (eqLogic, cmd, scenario, scenarioExpression) plutôt qu'un UNION.
Résultats structurés par catégorie, limit applicable par catégorie. Plus lisible et testé
individuellement via `sanitize_rows` par table.

**`list_files` dans `_core/logs.py` (pas dans `tools/logs.py`)**  
La logique de scan du système de fichiers appartient à `_core/` (accès bas niveau).
`tools/logs.py` délègue à `_core/logs.list_files()`, cohérent avec le pattern existant
(`tail_log` → `_core/logs.tail`).

**`tools/logs.py` — pas de `apikey` parameter**  
F5 est 100% MySQL + fichiers. Aucun enrichissement API JSON-RPC requis pour ces tools.
Cohérent avec la décision de routage D4bis.6.
